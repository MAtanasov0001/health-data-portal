# Портал за отворени данни в здравеопазването — посока към v0.2

## Контекст

Продължение на проекта **Диагноза България**. Целта сега е обща, между-институционална
инициатива за **Портал за отворени данни в здравеопазването**, институционално хостван от
**МИДТ** (без техен технически капацитет — разработката е общностна, вкл. потребителя).
Получена е концепция **v0.1** (силен документ). Тази сесия е *питане + мислене*, не
имплементация: потребителят иска (1) да потвърдя, че мога да работя **синхронно извън
OpenKBS** (споделен GitHub, друг домейн/хост), и (2) заедно да надградим v0.1 → v0.2, в която
аз излагам своите планове и виждания. Детайлните отговори за заинтересовани страни и визия
идват в следващото съобщение на потребителя.

---

## 1. Основният въпрос: да, мога да работя изцяло извън OpenKBS

Отговорът е недвусмислено **да** — и това *не е* компромис, а съвпада точно с принципите на
самия документ (раздел 16: *cloud-agnostic, преносимост, без обвързване с доставчик*).

**Как работя извън OpenKBS:**
- Каноничният **източник на истина е споделено Git хранилище** (GitHub/GitLab/state Git). Аз
  съм просто един контрибутор: чета, редактирам файлове, правя commit, push, отварям PR — по
  същия начин, независимо от хост.
- Всички доставки са **обикновени файлове в репото**: код (Next.js/Python), IaC
  (Terraform/Ansible), CI/CD конфигурации, DCAT-AP профил, документация. Нищо не зависи от
  OpenKBS-специфичен API. Стекът в раздел 14 е изцяло портируем.
- Продукцията върви там, където реши МИДТ (Държавен хибриден частен облак) — репото и
  контейнерите се внедряват еднакво навсякъде.

**Ролята на OpenKBS (по избор, препоръчително):** бърза *staging/демо* среда. Мога за кратко
да вдигна работещ прототип тук (статичен сайт + функция + Postgres + AI proxy), за да е
осезаема концепцията пред институциите — но кодът остава портируем в GitHub и демото е
изхвърляемо. Това е ускорител, не зависимост.

## 2. Препоръчан модел на съвместна работа (между-институционален)

- **Monorepo** с ясни модулни граници (`/frontend`, `/catalog`, `/ingestion`, `/infra`,
  `/docs`, `/spec`). Отговаря на раздел 12 (непоследователна архитектура → дефинирани граници).
- **Trunk-based + PR ревю**; по-строг режим за компонентите от раздел 13.1 (CODEOWNERS).
- **CI/CD с портите за сигурност от раздел 15 от ден 0** — преди първия ред продуктов код
  (както е и Фаза 0). Това е мястото, където guardrail-ите срещу vibe-coding се
  автоматизират, а не се разчита на дисциплина.
- **ADR** записи за проследимост на решенията (раздел 12/15).

## 3. Моят прочит на v0.1 — силни страни

Документът вече е на високо ниво: хибриден модел (harvest+snapshot по подразбиране, live
proxy по изключение), правилно позициониране спрямо EHDS (допълващ, не конкурент), DCAT-AP
3.0.1 + CKAN интероперативност с data.egov.bg, честен и конкретен vibe-coding риск-регистър,
ясна граница „какво НЕ се vibe-code-ва". Малко концепции за отворени данни стигат до това ниво
на зрялост.

## 4. Какво бих изострил/добавил за v0.2 (за обсъждане)

1. **Стратегическа развилка за каталожното ядро** (най-голямото решение):
   - (а) *Приемане/fork на CKAN* — както data.egov.bg; зряло, DCAT-AP native (препоръка на v0.1).
   - (б) *Лек собствен DCAT-AP каталог* — по-малко тежест, но повече vibe-coded повърхност в
     чувствителна зона (документът вече предупреждава срещу това).
   - (в) *Федериране В data.egov.bg* — здравният портал като тематична надстройка/харвест
     източник, вместо отделен портал. Различно позициониране на цялата инициатива.
2. **Пилотен набор от ден 1** — конкретен dataset (вкл. данни/поуки от Диагноза България), за
   да е end-to-end осезаема тръбата harvest → disclosure control → каталог → API, вместо само
   абстракция.
3. **Контролът на разкриването (small-cell suppression)** — да се извади като първокласен,
   независим, НЕ-vibe-coded модул с ясни правила (минимален размер на клетка, реидентификация
   при съчетаване), защото е ядрото на правната защита.
4. **Управление на данните (governance)** — кой е администратор/контролер, лицензен модел
   (CC BY 4.0?), DPIA собственик, процес по одобрение (раздел 17) — да се конкретизира с роли
   и хора.
5. **Реалистичен път при липса на API от титулярите** — раздел 19 го маркира; за v0.2 да се
   опише файлово подаване (CSV/Excel + метаданни) като пълноценна временна тръба, не бележка.
6. **Ясна карта: коя точно повърхност е vibe-coded** — обвързана с CI портите, за да е
   одитируемо решение, а не намерение.

## 5. Ключови развилки

**Потвърдено:**
- ✅ **Хостинг/съвместна работа: публичен GitHub от ден 1.** Open-source от старта — максимална
  прозрачност, подходящо за общностна между-институционална инициатива. Аз работя през PR-и.
  Следствия за v0.2: публичен репо изисква от ден 0 → сканиране за тайни (никакви .env в git,
  vault), чист secrets-hygiene, LICENSE + CONTRIBUTING + CODEOWNERS, публично видими CI порти.

**Чакащи (за следващото съобщение на потребителя, заедно с визия/страни):**
- Каталожно ядро: CKAN adopt / собствено / федериране в data.egov.bg.
- Дали да вдигна работещ прототип на OpenKBS паралелно (бързо демо), при GitHub като истина.
- Заинтересовани страни, визия, идея за портала.

## 6. Следващи стъпки

1. Потребителят потвърждава развилките по-горе + дава визия/страни.
2. Записвам **v0.2** на концепцията (надграждам структурата на v0.1 с горните допълнения).
3. Едва тогава — детайлна тех. спецификация и (по избор) прототип.

> Забележка: това е стратегическа/консултативна фаза — още няма код за имплементация. Този

### Реални отворени въпроси (за да СТАРТИРАМ производство на код)

1. **Капацитет за ревю — #1 риск по самия документ (15.1, 21).** Аз съм авторът на ~80% от
   кода, вкл. чернови по режим C. Не мога да само-одобрявам режим B (инженер ≠ автор) и режим C
   (експерт по сигурност + двама одобряващи). Кой играе ревюиращ? Това определя реда: първо
   произвеждам режим A/B, докато се осигурят ревюиращи за C.
2. **Как процедираме сега спрямо Фаза 0.** git.egov.bg + вписванията са институционални и бавни.
   Мога да започна pre-Phase-0 основа в GitHub работно репо (скеле + CI порти + DCAT-AP контракт
   + файлов пилот), която после става огледалото — или чакаме Фаза 0.
3. **Лиценз (EUPL-1.2 vs MIT)** — фиксира се преди първия публичен комит (13.2). Блокер за старт.
4. **Потвърждение на вариант А (CKAN).** Ако да — строя Next.js frontend срещу API контракта и
   вдигам CKAN зад каталога; frontend + контракт мога да започна дори преди CKAN (мок контракт).
5. **Пилотен набор.** Предлагам файлов набор от Диагноза България сега → пълна тръба end-to-end
   (приемане → схема → контрол на разкриването → каталог → API → frontend), без да чакаме API на
   титуляри.
6. **Обхват на анонимизацията в портала (техническо уточнение).** Титулярите дават вече
   анонимизирани/агрегирани данни; порталът добавя *независим контрол на разкриването*
   (small-cell suppression). Въпрос: получава ли порталът изобщо запис-ниво идентификатори, за да
   прилага еднопосочната хеш+сол функция по чл. 15, ал. 4 — или ролята му е чисто агрегатно
   потискане? Определя дали строя криптографски компонент изобщо. Иска уточнение с титуляри/юрист.

## СТАРТОВ ПЛАН (pre-Phase-0 основа) — потвърден

**Потвърдено от потребителя:** старт сега в GitHub работно репо (после огледало на git.egov.bg).
**Приети начални стойности (по препоръка на v2.0, подлежат на смяна):** вариант А (CKAN + собствен
frontend); пилот = файлов набор от Диагноза България; лиценз MIT (работно, до становище на юрист).

**Принцип на реда (заради риск №1 — капацитет за ревю):** произвеждам първо **режим A/B**;
**режим C** (сигурност/идентичност/интеграционен слой/анонимизация) се отлага, докато не се
осигурят ревюиращи — за да не се трупа непрегледан критичен код (15.1). Конвейерът с портите се
изгражда **преди първия продуктов ред** (15.2). Доставка: генерирам файлове тук на порции,
потребителят ги качва в GitHub.

### Структура на монорепото
```
README.md  LICENSE(MIT)  CONTRIBUTING.md  CODEOWNERS  SECURITY.md  .gitignore(без .env)
.github/workflows/ci.yml   # блокиращи порти
/docs/            # архитектура; профил DCAT-AP
/spec/dcat-ap/    # метаданен профил + контролирани речници (EuroVoc, NUTS)
/frontend/        # Next.js (React, TS) — режим A: 5 страници + шаблон за набор
/catalog/         # CKAN (вариант А) — docker-compose локално
/ingestion/       # Python (FastAPI+Pydantic) — приемане, схема, контрол на разкриването, версии
/api/             # публично четящо API (OpenAPI 3.1) — DCAT/RDF + CKAN-съвместимо
/infra/           # Terraform/Ansible; docker-compose за локална среда
/pilot/           # файлов набор от Диагноза България + метаданни
```

### Първи вертикален срез (end-to-end, само режим A/B)
1. **Скеле + правни файлове:** LICENSE (MIT), CONTRIBUTING, CODEOWNERS, SECURITY.md (отговорно
   разкриване), .gitignore без тайни.
2. **CI/CD порти ПЪРВО (15.2):** lint, type-check, unit/integration/e2e (скелетни), SAST,
   анализ на зависимости + SBOM, **сканиране за тайни (блокиращо от ден 0)**, a11y (EN 301 549).
   Плюс контролен списък за нормативно съответствие (15.3) като PR шаблон.
3. **DCAT-AP 3.0.1 профил** — полета + контролирани речници, машинно четимо.
4. **Приемане (Python):** пилотен файл → Pydantic схемна валидация → контрол на разкриването
   (min размер на клетка / small-cell suppression) → неизменяем версиониран снапшот с контролна
   сума и времеви печат по ISO 8601 / UTC+2 (чл. 46).
5. **Публично API (OpenAPI 3.1):** списък/детайл на набори, филтри+пагинация (МЕ90), GET на
   уникален адрес (МЕ72), версия /v1, content negotiation CSV/JSON/XLSX/RDF, DCAT/RDF +
   CKAN-съвместими крайни точки за харвестване.
6. **Frontend (Next.js):** петте страници + шаблон за набор, консумира публичното API, BG/EN,
   WCAG 2.2 AA, семантичен HTML5 + schema.org + Open Graph (МЕ76–78), SSR за SEO.
7. **CKAN (вариант А)** зад каталога, локално чрез docker-compose; API/frontend говорят с него.

### Отложено до наличие на ревюиращи за режим C
Ядро на сигурността, автентикация/оторизация (национална е-идентификация + MFA), хранилище за
тайни, жив proxy (SSRF-устойчив), конектори към API на титуляри, крипто за анонимизация (чака
уточнение по въпрос 6 за обхвата).

### Проверка (как тествам среза локално)
- `npm run lint && npm run type-check && npm test` за frontend; a11y тест минава.
- Приемане: пускам пилотния файл през тръбата → версиониран снапшот + отхвърляне при невалидна схема.
- Вдигам docker-compose (CKAN + API + frontend), отварям петте страници + страница на набор в
  браузър (agent-browser); golden path: търсене → филтър → набор → изтегляне → API пример;
  проверявам пагинация и уникален GET URL за всеки изглед.
- CI конвейерът минава зелено на PR преди сливане.
