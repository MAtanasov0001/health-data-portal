# Пилотен набор — Диагноза България (файлово подаване)

Този каталог съдържа **пилотния набор от данни** за портала, подаден по **файловия път**
(CSV + метаданни), без да се чака API от титуляр. Целта е тръбата да е осезаема end-to-end:

```
файл (CSV + метаданни) → схемна валидация → контрол на разкриването → неизменяем версиониран
снапшот → каталог (DCAT-AP) → публично API → frontend
```

## Съдържание

| Файл | Предназначение |
|------|----------------|
| [`hospitalizacii-po-oblast-2023/source.csv`](hospitalizacii-po-oblast-2023/source.csv) | Суров агрегиран вход (примерни/синтетични данни) |
| [`hospitalizacii-po-oblast-2023/metadata.yaml`](hospitalizacii-po-oblast-2023/metadata.yaml) | Метаданни на набора (влизат в DCAT-AP профила) |
| [`hospitalizacii-po-oblast-2023/schema.yaml`](hospitalizacii-po-oblast-2023/schema.yaml) | Схема на колоните за валидация при приемане |

## Важно за данните

Стойностите тук са **синтетични/примерни** и служат само за демонстрация на тръбата. Те **не
са реални здравни данни**. Реалните набори идват от титулярите вече агрегирани; порталът добавя
**независим контрол на разкриването** (small-cell suppression) върху агрегатите.

Стойности под прага за минимален размер на клетка се маркират в суровия вход като `<5`, за да
се провери, че приемната тръба ги разпознава и потиска коректно (без да позволява възстановяване
чрез редови/колонни суми).

## Стартиране на пилота през тръбата

```bash
cd ingestion
pip install -e ".[dev]"
python -m ingestion.pipeline run ../pilot/hospitalizacii-po-oblast-2023
# → създава версиониран снапшот в ingestion/snapshots/ с контролна сума и времеви печат (UTC+2)
```

## Реален набор — НЗОК: дейности по клинични пътеки/процедури, 2025

`deynosti-kp-nzok-2025/` е **реален** набор от титуляря НЗОК (данните зад Диагноза България),
подаден като **експорти** (12 месечни `.xlsx`), не през API. Тъй като реалните данни идват
като файлове, приемането има допълнителна **зареждаща стъпка** преди тръбата:

```
12 месечни .xlsx (експорт на НЗОК)
   → зареждащ адаптер (годишен агрегат, нормализация)  →  source.csv
   → схемна валидация → контрол на разкриването → снапшот → DCAT-AP → API → frontend
```

| Файл | Предназначение | В git? |
|------|----------------|--------|
| `deynosti-kp-nzok-2025/schema.yaml` | Схема на колоните + правила за разкриване | ✅ да |
| `deynosti-kp-nzok-2025/metadata.yaml` | Метаданни (DCAT-AP) | ✅ да |
| `deynosti-kp-nzok-2025/source.csv` | Производен ПРЕДИ-потискане вход | ❌ **не** (gitignore) |
| суровите `.xlsx` експорти | Вход от титуляря | ❌ **не** (в `data/`) |

> **Защо source.csv не се комитва:** той съдържа малките клетки (ЗОЛ<5), които контролът на
> разкриването потиска. Публикуването му в публичното репо би обезсмислило защитата. Комитват
> се само spec-файловете; входът и снапшотът се **регенерират** от суровите експорти.

### Стъпки

```bash
# 0) суровите месечни .xlsx се разполагат в gitignore-нат work dir, напр. data/nhif-activities-2025-raw/
# 1) зареждащ адаптер: 12-те месеца → годишен source.csv
python -m ingestion.loaders.nhif_activities \
    data/nhif-activities-2025-raw pilot/deynosti-kp-nzok-2025/source.csv
# 2) тръбата: source.csv → снапшот с контрол на разкриването
cd ingestion && python -m ingestion.pipeline run ../pilot/deynosti-kp-nzok-2025
```

### Особености на реалните данни (нормализирани от адаптера)

- **Непоследователен формат на РЗОК между месеците:** м. 01–05 подават „код + име"
  (`01 Благоевград`), м. 06–12 само кода (`01`). Адаптерът нормализира до канонично „код + име",
  за да не се раздвоява един регион на два ключа.
- **Годишен агрегат:** сумиране на 12-те месеца по (РЗОК, ЛЗБП, КП/АПр/КПр, осн. дг, вторична дг).
- **Две мерки:** брой ЗОЛ (обвързваща за разкриването) и брой случаи (свързана — скрива се синхронно).
- **Гранулярност/потискане:** на пълна категорийна гранулярност ~85% от клетките са под праг
  (ЗОЛ<5) и се потискат — правно най-сигурният избор; публикуват се ~50 хил. реда.

## Реален набор — НЗОК: болнични лекарства (стар `.xls`), 2025

`deynosti-lekarstva-nzok-2025/` е втори **реален** набор от НЗОК („Справка 5 — ПЛС2":
противотуморни продукти и такива за вродени коагулопатии). Тук експортите идват в **стария
двоичен формат `.xls`** (OLE2/BIFF8), а не в `.xlsx` — затова има отделен зареждащ адаптер и
собствен четец, но същата приемна тръба:

```
12 месечни .xls (стар OLE2/BIFF8 експорт на НЗОК)
   → четец ingestion/xls.py (само stdlib) + зареждащ адаптер (годишен агрегат)  →  source.csv
   → схемна валидация → контрол на разкриването → снапшот → DCAT-AP → API → frontend
```

| Файл | Предназначение | В git? |
|------|----------------|--------|
| `deynosti-lekarstva-nzok-2025/schema.yaml` | Схема на колоните + правила за разкриване | ✅ да |
| `deynosti-lekarstva-nzok-2025/metadata.yaml` | Метаданни (DCAT-AP) | ✅ да |
| `deynosti-lekarstva-nzok-2025/source.csv` | Производен ПРЕДИ-потискане вход | ❌ **не** (gitignore) |
| суровите `.xls` експорти | Вход от титуляря | ❌ **не** (в `data/`) |

### Стъпки

```bash
# 0) суровите месечни .xls в gitignore-нат work dir, напр. data/nhif-hospital-drugs-2025-raw/
# 1) зареждащ адаптер: 12-те .xls месеца → годишен source.csv
python -m ingestion.loaders.nhif_hospital_drugs \
    data/nhif-hospital-drugs-2025-raw pilot/deynosti-lekarstva-nzok-2025/source.csv
# 2) тръбата: source.csv → снапшот с контрол на разкриването
cd ingestion && python -m ingestion.pipeline run ../pilot/deynosti-lekarstva-nzok-2025
```

### Особености на реалните данни (нормализирани от адаптера)

- **Стар двоичен `.xls` (OLE2/BIFF8):** `ingestion/xls.py` е тесен четец **само на стандартна
  библиотека** (без `xlrd`/`openpyxl`) — приемната тръба остава без външни зависимости.
  Покрива точно каквото експортите ползват: OLE2 съставен файл (DIFAT/FAT, директория, голям и
  мини поток) + BIFF записите `SST`/`LABELSST`/`LABEL`/`RK`/`MULRK`/`NUMBER`.
- **РЗОК се извежда от РЦЗ:** файлът дава само регистрационния код на лечебното заведение;
  адаптерът извлича региона от **първите две цифри** на РЦЗ и го нормализира до „код + име".
- **Годишен агрегат:** сумиране на 12-те месеца по 13-те категорийни измерения (ЛЗ, ATC, INN,
  НЗОК код, продукт, форма, МКБ, …).
- **Три мерки:** брой ЗОЛ (обвързваща за разкриването) + брой опаковки и реимбурсна сума
  (свързани — скриват се синхронно, когато ЗОЛ клетката е потисната).
- **Гранулярност/потискане:** на пълна гранулярност ~56% от клетките са под праг (ЗОЛ<5) и се
  потискат; публикуват се ~50 хил. реда (реидентификацията при малка кохорта е реална, затова
  свързаните мерки се нулират в синхрон).
