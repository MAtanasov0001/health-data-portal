# Архитектура на сигурността

Този документ описва пласта на сигурността на портала: какво е изградено, къде живее, дали е
**активно** или е **готов seam** (изключено по подразбиране), и срещу какви заплахи защитава.
Придружава [`review-model.md`](review-model.md) и решението на титуляра за изпреварващо създаване
на режим-C код (виж там, раздел „Ред на разработката“).

Принципи: secure-by-default, fail-closed, най-малки права, нула тайни в хранилището, всичко
конфигурируемо през околната среда, всеки чувствителен компонент — pluggable зад ясен интерфейс.

---

## Инвентар на компонентите

| Компонент | Път | Режим | Състояние | Роля |
|-----------|-----|-------|-----------|------|
| App-security пласт | `api/api/security/middleware.py` | C | **активен** | защитни хедъри, CORS, rate limit, лимит на тялото, безопасни грешки |
| Валидация на вход | `api/api/security/validation.py` | C | **активен** | отхвърля обхождане на пътища/неканонични идентификатори (400) |
| Клиентски IP | `api/api/security/net.py` | C | **активен** | доверени прокси hop-ове; игнорира X-Forwarded-For по подразбиране |
| Доставчик на тайни | `api/api/security/secrets.py` | C | seam (env бекенд) | `SecretProvider`; продукционен мениджър — plug-point |
| Идентичност/оторизация | `api/api/auth/` | C | seam (**disabled**) | pluggable IdP, route guard; е-ID+MFA plug-point |
| SSRF-устойчив клиент | `api/api/security/ssrf.py` | C | seam (несвързан) | безопасно изходящо теглене за харвест/жив proxy |
| Псевдонимизация | `ingestion/ingestion/anonymization/` | C | seam (**off**) | еднопосочна хеш+сол (чл. 15, ал. 4) |
| Контрол на разкриването | `ingestion/ingestion/disclosure.py` | B/C | **активен** | small-cell suppression (първично + вторично) |

„seam“ = кодът съществува и е тестван, но е изключен по подразбиране и не е закачен за работещ
маршрут; изисква пълния режим-C одит преди активиране (виж контролния списък по-долу).

## Модел на заплахите (актив → заплаха → контрола)

| Актив / повърхност | Заплаха | Контрола |
|--------------------|---------|----------|
| Публичното четящо API | DoS чрез заявки | rate limit (per-IP, sliding window), fail-open при вътрешна грешка на лимитера, за да не спре достъпът до отворени данни |
| Публичното API | изчерпване на ресурси с голямо тяло | лимит на `Content-Length`/тялото (413); API-то е GET-only |
| Отговори на API | изтичане на вътрешна информация при грешка | уловени необработени грешки → 500 с `correlation_id`, без стектрейс/детайли навън |
| Идентификатори в пътя | обхождане на пътища / инжекция | канонична валидация (`^[a-z0-9]+(?:-[a-z0-9]+)*$`, дължина ≤ 100) |
| Браузър клиенти | clickjacking / MIME sniffing / изтичане на referrer | `X-Frame-Options: DENY`, строг CSP, `nosniff`, `Referrer-Policy: no-referrer`, COOP |
| Транспорт | понижаване към HTTP | HSTS (зад терминиращ TLS), configurable |
| Тайни | тайна в кода/хранилището/логовете | доставчик на тайни от env/мениджър; `redact()`; сканиране за тайни като блокираща CI порта |
| Лични данни (при запис-ниво вход) | реидентификация / свързване между набори | еднопосочна HMAC-псевдонимизация с таен pepper + контекстна сол (несвързваемост между набори) |
| Агрегати | реидентификация от малки клетки | small-cell suppression (първично + вторично потискане) |
| Изходящи връзки (бъдещ харвест/proxy) | SSRF към вътрешна мрежа/облачен metadata | HTTPS-only, проверка на всички разрешени IP-та, пиниране към проверен IP, без redirect, таймаут + таван на размера, по избор host allowlist |
| Защитени операции (бъдещи) | неоторизиран достъп / липса на MFA | fail-closed route guard; бекенд `disabled` по подразбиране; изискване за обхвати и втори фактор |

## Конфигурация (околна среда)

Всички настройки имат сигурни стойности по подразбиране; средата само разхлабва там, където е нужно.

| Променлива | По подразбиране | Описание |
|------------|-----------------|----------|
| `OHDP_CORS_ORIGINS` | (празно) | разрешени CORS произходи (или `*` за напълно отворени) |
| `OHDP_RATE_LIMIT_ENABLED` | `true` | включва ограничаването на честотата |
| `OHDP_RATE_LIMIT_REQUESTS` / `OHDP_RATE_LIMIT_WINDOW` | `120` / `60` | заявки на прозорец (сек) |
| `OHDP_TRUSTED_PROXY_HOPS` | `0` | брой доверени обратни проксита (за клиентски IP) |
| `OHDP_MAX_BODY_BYTES` | `65536` | таван на размера на тялото |
| `OHDP_HSTS_ENABLED` / `OHDP_HSTS_MAX_AGE` | `true` / `63072000` | HSTS зад TLS |
| `OHDP_SECRET_BACKEND` | `env` | бекенд за тайни (`env`; др. — plug-point) |
| `OHDP_SECRET_*` | — | стойности на тайни (напр. `OHDP_SECRET_PEPPER`) |
| `OHDP_AUTH_BACKEND` | `disabled` | бекенд за идентичност (`disabled` / `localdev`; е-ID — plug-point) |
| `OHDP_AUTH_DEV_TOKENS` | — | статични токени за разработка (само `localdev`) |

## Контролен списък за активиране на режим-C компонент

Преди който и да е от seam-овете да бъде включен в продукция:

1. Експертен ревю по сигурност + двама одобряващи (режим C).
2. Външен тест за проникване на компонента и неговата интеграция.
3. Реален доставчик на тайни (не env бекендът) за всякакви продукционни тайни.
4. За идентичност: свързан е-ID/OIDC доставчик + MFA; преглед на обхватите на всеки защитен маршрут.
5. За жив proxy/харвест: потвърден host allowlist, egress-филтриране на мрежово ниво, ревю на
   SSRF-контролите срещу актуалния модел на заплахите.
6. За псевдонимизация: потвърден обхват с титуляри/юрист (получава ли порталът запис-ниво данни),
   ротация/съхранение на pepper-а, DPIA.
