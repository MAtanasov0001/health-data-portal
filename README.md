# Портал за отворени данни в здравеопазването

**Open Health Data Portal (Bulgaria)** — публичен, машинночетим достъп до анонимизирани и
агрегирани здравни данни, предоставяни от техните титуляри (Министерство на здравеопазването,
НЗОК, „Информационно обслужване“ АД и др.).

Инициатива на **Министерството на иновациите и дигиталната трансформация (МИДТ)**, изпълнявана
като **отворена разработка** съвместно с ИТ общността в България. Продължение на гражданския
проект *Диагноза България* — от независим граждански одит към устойчива обществена инфраструктура.

> Каноничен архитектурен документ: [`docs/concept-v2.0.md`](docs/concept-v2.0.md) (концепция
> v2.0, приведена в съответствие с **НОИИСРЕАУ**). Този код реализира стартовата (pre-Phase-0)
> основа.

## Какво е това и защо

Обществените пари за здравеопазване са публични — прозрачността за тяхното изразходване все още
не е. Данните *ги има* (в отчети, регистри, обществени поръчки), но са пръснати така, че никой не
ги сравнява или събира на едно място в машинночетим вид.

Този портал прави точно това: събира разходите и дейностите в здравеопазването, прекарва ги през
**независим контрол на разкриването** и ги публикува като **отворени, версионирани и оперативно
съвместими** набори — достъпни както за хора (уебпортал), така и за машини (API, DCAT-AP).

Целевите потребители са граждани, журналисти, изследователи, лечебни заведения и институции.
Прозрачността е предпоставка за доверие и за по-добри решения в системата.

## Възможности

- **API-first (чл. 41).** Всеки набор е достъпен програмно на стабилен адрес, с версия `/v1` и
  поддръжка минимум 24 месеца.
- **Договаряне на формат.** Един и същ набор като **CSV, JSON, XLSX и RDF** (Turtle/JSON-LD).
- **DCAT-AP 3.0.1 + CKAN.** Метаданните са готови за автоматично харвестване от `data.egov.bg`
  и `data.europa.eu`; CKAN-съвместими крайни точки (`/api/3/action/...`).
- **Контрол на разкриването.** Потискане на малки клетки (small-cell suppression) срещу
  реидентификация при съчетаване на данни.
- **Неизменяеми версионирани снапшоти** с контролна сума SHA-256 и времеви печат по ISO 8601
  (UTC+2, чл. 46) + одитна следа (МЕ64).
- **Двуезичен, достъпен уебпортал** (BG/EN, WCAG 2.2 AA, SSR за SEO).
- **Secure-by-default, fail-closed** пласт за сигурност (режим C), изключен по подразбиране до
  експертен одит.
- **Публично от ден 1** — целият код, правила и история на решенията са отворени.

## Статус

🟡 **Pre-Phase-0 основа.** Изграден е вертикален срез от **режим A/B** (вж.
[`docs/review-model.md`](docs/review-model.md)): слой за приемане/ETL, публично четящо API,
метаданен профил и уебпортал.

Пластът на сигурността (**режим C**) — защитни хедъри/rate limit, доставчик на тайни, идентичност
и оторизация, SSRF-устойчив клиент, еднопосочна псевдонимизация (чл. 15, ал. 4) — вече е **създаден
предварително** по решение на титуляра (2026-07-23). Целият режим-C код е **изключен/неактивен по
подразбиране** и подлежи на експертен одит + тест за проникване преди активиране. Инвентар, модел на
заплахите и контролен списък за активиране: [`docs/security-architecture.md`](docs/security-architecture.md).

## Архитектура

Хибриден модел на данните: **приемане и кеширане по подразбиране** (harvest + версиониран
снапшот), **жив достъп само по изключение** през строго контролиран интеграционен слой.
Публичните потребители четат от локалното копие, никога пряко от системите на титулярите.

```
Титуляри (API/файл) ──▶ [Интеграционен слой] ──▶ [Приемане/ETL + контрол на разкриването]
                                                          │
                                          версиониран снапшот (immutable)
                                                          ▼
                            [Каталожно ядро: CKAN + DCAT-AP]  ◀──▶  [Публично API]
                                                          ▲                 ▲
                                                          └──── [Frontend (Next.js)] ──── потребители
```

**Принципи на потока:**

1. **Събиране** — данните постъпват от титулярите като файлове или през API, вече анонимизирани
   и агрегирани.
2. **Контрол на разкриването** — малките клетки се потискат независимо, преди публикуване.
3. **Снапшот** — резултатът се замразява като неизменяема версия с контролна сума и времеви печат.
4. **Каталог** — наборът се описва по DCAT-AP и се вписва в CKAN (вариант А).
5. **Публикуване** — API-то и уебпорталът четат само от снапшота.

## Технологичен стек

**Frontend:** Next.js 15 (App Router, React, TypeScript), SSR за SEO, self-hosted Sofia Sans,
без клиентски зависимости за визуализациите (чист SVG).
**Каталог:** CKAN 2.10 + Solr.
**Backend/ETL:** Python (FastAPI + Pydantic).
**Данни:** PostgreSQL 16 · Redis 7 · S3-съвместимо обектно хранилище.
**Инфраструктура:** Reverse proxy + WAF · Docker · Terraform/Ansible.

Изборът е обоснован в [`docs/concept-v2.0.md`](docs/concept-v2.0.md), раздел 16.

## Локална разработка

**Изисквания:** Docker + Docker Compose (за пълния стек); Node.js 20+ (frontend); Python 3.11+
(API/ETL).

### Пълна локална среда (CKAN + API + frontend + Postgres + Redis + Solr)

```bash
docker compose -f infra/docker-compose.yml up
```

Услуги в compose: `db` (Postgres), `redis`, `solr`, `ckan`, `api`, `frontend`, `catalog-sync`.

### Само frontend

```bash
cd frontend
npm install
npm run dev            # http://localhost:3000  (по подразбиране API на 127.0.0.1:8000)

npm run lint           # ESLint
npm run type-check     # tsc --noEmit
npm test               # Vitest (вкл. a11y)
```

### Само API

```bash
cd api
pip install -e .
uvicorn api.main:app --reload      # http://127.0.0.1:8000  ·  /docs за Swagger UI
```

Полезни променливи на средата: `OHDP_SNAPSHOTS` (път до снапшотите),
`OHDP_RATE_LIMIT_ENABLED`, `OHDP_BASE_URL`.

### Тръба за приемане (пилотен набор)

CLI-то е `ohdp-ingest` (конзолен скрипт от пакета `ingestion`):

```bash
cd ingestion
pip install -e .

# Пусни набор през тръбата: валидация по схема → контрол на разкриването → версиониран снапшот
ohdp-ingest run ../pilot/deynosti-kp-nzok-2025
# → снапшот в ingestion/snapshots/, контролна сума, брой потиснати клетки, DCAT-AP, одит (МЕ64)
```

Всеки пилотен набор е директория със `source.csv`, `schema.yaml` и `metadata.yaml`.
> ⚠️ `source.csv` за реалните набори **не се комитва** (съдържа малките клетки, които контролът
> потиска) — вижте [`pilot/README.md`](pilot/README.md).

## Публично API

Базов адрес по подразбиране за локална среда: `http://127.0.0.1:8000`.
Интерактивна документация: **`/docs`** (Swagger UI) · схема: **`/openapi.json`** (OpenAPI 3.1).

| Метод | Път | Описание |
|-------|-----|----------|
| `GET` | `/v1/health` | Проверка на състоянието |
| `GET` | `/v1/datasets` | Списък с набори (пагиниран: `?page=&page_size=`) |
| `GET` | `/v1/datasets/{id}` | Детайл на набор (DCAT-AP метаданни) |
| `GET` | `/v1/datasets/{id}/versions` | Всички версии на набор |
| `GET` | `/v1/datasets/{id}/summary` | Агрегат по измерение (за визуализации) |
| `GET` | `/v1/datasets/{id}/data` | Данни с **договаряне на формат** (`Accept:`) |
| `GET` | `/v1/datasets/{id}/data.csv` | Дистрибуция CSV |
| `GET` | `/v1/datasets/{id}/data.json` | Дистрибуция JSON (пагинирана) |
| `GET` | `/v1/datasets/{id}/data.xlsx` | Дистрибуция XLSX (МЕ34) |
| `GET` | `/v1/datasets/{id}/dcat.jsonld` | Метаданни на набора (JSON-LD) |
| `GET` | `/v1/datasets/{id}/dcat.ttl` | Метаданни на набора (RDF/Turtle) |
| `GET` | `/v1/catalog.jsonld` | DCAT-AP каталог (JSON-LD) |
| `GET` | `/v1/catalog.ttl` | DCAT-AP каталог (RDF/Turtle) |
| `GET` | `/api/3/action/package_list` | CKAN-съвместимо харвестване (списък) |
| `GET` | `/api/3/action/package_show` | CKAN-съвместимо харвестване (детайл) |

### Примери

```bash
# Списък с набори (пагиниран)
curl "http://127.0.0.1:8000/v1/datasets?page=1&page_size=20"

# Детайл на набор
curl "http://127.0.0.1:8000/v1/datasets/hospitalizacii-po-oblast-2023"

# Данни в различни формати
curl "http://127.0.0.1:8000/v1/datasets/hospitalizacii-po-oblast-2023/data.csv"
curl "http://127.0.0.1:8000/v1/datasets/hospitalizacii-po-oblast-2023/data.json"

# Договаряне на формат чрез Accept
curl -H "Accept: application/xlsx" \
  "http://127.0.0.1:8000/v1/datasets/hospitalizacii-po-oblast-2023/data"

# DCAT-AP каталог (за харвестъри) и CKAN-съвместимо
curl "http://127.0.0.1:8000/v1/catalog.jsonld"
curl "http://127.0.0.1:8000/api/3/action/package_list"
```

## Frontend (уебпортал)

Next.js (App Router) с двуезична маршрутизация `/(bg|en)/...`:

| Път | Страница |
|-----|----------|
| `/{lang}` | Начална — hero, метрики от живите данни, интерактивни графики, „как работи", мисия |
| `/{lang}/naborite` | Списък с набори |
| `/{lang}/naborite/{id}` | Детайл на набор — метаданни, визуализация, преглед на данните, изтегляне, API |
| `/{lang}/dokumentacia` | Техническа документация (API, формати, харвестване) |
| `/{lang}/za-portala` | За портала (произход, управление, стандарти) |
| `/{lang}/kontakti` | Контакти |

**Бранд:** MEDATA (работно име) + подзаглавие „Портал за отворени данни в здравеопазването";
зелена палитра с приемственост към НЗИС (his.bg); шрифт Sofia Sans (self-hosted, без външни
заявки). Достъпност: WCAG 2.2 AA, семантичен HTML5, schema.org + Open Graph, SSR за SEO.

**Живо демо / staging:** реалният `/frontend` се експортира статично (`STATIC_EXPORT=1`) и се
деплойва на OpenKBS (S3 + CloudFront) само за визуален преглед на дизайна; данните са замразени
към момента на билда. Продукцията върви като SSR. Staging-билдът (`/site/`) е изхвърляем и **не
се комитва** — каноничният frontend е в `/frontend`.

## Непрекъсната интеграция (CI)

Портите се изпълняват от [`.github/workflows/ci.yml`](.github/workflows/ci.yml) и са **блокиращи
от ден 0** (преди първия продуктов ред):

- **Сканиране за тайни** (gitleaks) — нула тайни в публичното репо.
- **Frontend** — ESLint · TypeScript · Vitest · a11y (EN 301 549).
- **Python** — ruff (lint+format) · mypy (строга типизация) · pytest · **валидатор на DCAT-AP
  профила**.
- **SAST** (Semgrep).
- **SCA + SBOM** (Trivy за уязвимости в зависимости/IaC; Syft за CycloneDX SBOM).
- **Нормативен контролен списък** — PR тялото трябва да съдържа контролния списък за съответствие.

## Сигурност

Пластът на сигурността е secure-by-default и fail-closed. Активни контроли на публичното API:
защитни HTTP хедъри (CSP, `nosniff`, `X-Frame-Options`), CORS по избор, ограничаване на честотата,
лимит за размер на тялото, безопасно обработване на грешки (без изтичане) и канонична валидация на
идентификаторите. Готови, но **изключени по подразбиране** seam-ове: доставчик на тайни, идентичност
+ MFA (route guard), SSRF-устойчив клиент за бъдещ харвест/жив proxy и еднопосочна псевдонимизация.

Пълен инвентар, модел на заплахите, конфигурация през околната среда и контролен списък за
активиране: **[`docs/security-architecture.md`](docs/security-architecture.md)**.
Разкриване на уязвимости: **[`SECURITY.md`](SECURITY.md)** (не в публични задачи).

## Модел на ревю

Тристепенен режим на ревю (заменя разделението vibe/не-vibe):

- **Режим A** — стандартно ревю (frontend, пилотни данни, документация).
- **Режим B** — усилено ревю (API, ETL, каталог, инфраструктура).
- **Режим C** — критично: сигурност, идентичност, интеграционен слой, анонимизация → експерт по
  сигурност + двама одобряващи + тест за проникване, преди активиране.

Подробности и CODEOWNERS-обвързване: [`docs/review-model.md`](docs/review-model.md).

## Стандарти и съответствие

- **DCAT-AP 3.0.1** + CKAN-съвместимо харвестване (`data.egov.bg`, `data.europa.eu`).
- **API-first** (чл. 41), версии `/v1` + поддръжка ≥ 24 мес., GET на уникален адрес (МЕ72),
  филтри + пагинация (МЕ90), договаряне на формат CSV/JSON/XLSX/RDF (МЕ34).
- **Времеви печати** по ISO 8601 / UTC+2 (чл. 46); **одитна следа** с 9 атрибута (МЕ64).
- **Контрол на разкриването** (потискане на малки клетки) срещу реидентификация.
- **Достъпност** WCAG 2.2 AA / EN 301 549; **SEO** — семантичен HTML5, schema.org, Open Graph
  (МЕ76–78).
- **Отворен лиценз** и публичен изходен код от ден 1.
